# 简历英文翻译 Skill

## 什么时候使用

当用户需要把中文简历、项目经历、工作经历、教育背景、求职材料翻译或润色成英文时，使用这个 Skill。

## 核心目标

把中文简历转成**地道、专业、求职导向的英文简历表达**，避免中式英语，同时保留项目成果、数量、影响和关键词。

## 固定风格要求

- 默认使用正式英文简历语气，简洁、有力、自然。
- 默认字体要求：Times New Roman。
- 不要逐字硬译中文。
- 不要写成夸张营销文案。
- 尽量使用英文简历常见动词：led, managed, coordinated, edited, produced, analyzed, developed, optimized, evaluated, supported 等。
- 保留并突出可量化成果，例如：20+ videos, 72 graded readers, 1,300 comments, 33,547 articles。
- 如果中文信息含糊，不要编造数据；可以保守表达或标注“需补充”。

## 工作流程

1. 先判断用户给的是：
   - 完整简历
   - 单段经历
   - 项目描述
   - 求职方向说明
2. 识别目标方向：
   - AI 产品 / AI 评测
   - 金融科技 / 商业分析
   - 内容产品 / 媒体行业
   - 学术 / 研究助理
   - 其他
3. 翻译时优先保留：
   - 学校、专业、时间
   - 公司/机构名称
   - 角色名称
   - 项目成果
   - 数据和规模
   - 技能关键词
4. 对每条经历尽量使用：
   - 动作动词 + 任务内容 + 方法/工具 + 结果/影响
5. 最后检查：
   - 是否有中式英语
   - 是否过度夸大
   - 是否时间/数字/机构名不一致
   - 是否适合英文简历阅读习惯

## 输出格式

根据任务选择：

### 如果用户要完整英文简历

输出：
1. 英文简历正文
2. 可选：修改说明
3. 可选：仍需用户确认的信息

### 如果用户只给一段经历

输出：
- 推荐英文版本
- 更强一点的版本
- 更保守一点的版本
- 用词说明

### 如果用户要 Word 文档

应生成 `.docx`，字体 Times New Roman，版式简洁，不要复杂装饰。

## 避免

- 不要把“参与”一律翻成 participated in。
- 不要把“负责”一律翻成 responsible for。
- 不要加入用户没说过的奖项、排名、成绩或成果。
- 不要把中文机构名乱翻，必要时保留拼音或官方英文名。
- 不要堆砌 buzzwords。
```

# 每日财经简报 Skill

## 什么时候使用

当用户要求“每日简报”“最近财经新闻”“AI/科技/财经公司动态”“选题线索”“写作线索”时，使用这个 Skill。

## 核心目标

不是简单罗列新闻，而是把新闻转化成**可理解、可记忆、可继续写作和研究的线索**。

## 默认关注范围

- AI 与科技公司
- 财经与公司动态
- 中国经济与产业政策
- 全球宏观经济、利率、通胀、资本市场
- 新闻传播/媒体行业中与平台、AI、商业模式有关的变化

## 固定输出结构

每次按这个结构输出：

1. **重要新闻**
   - 发生了什么
   - 大白话解释
   - 为什么重要
   - 和用户的写作/研究有什么关系

2. **趋势信号**
   - 从新闻背后提炼 3-5 个大趋势
   - 避免只说“值得关注”
   - 要说明趋势怎么形成、可能影响谁

3. **可写选题角度**
   - 给出可直接发展成文章/报告/选题的角度
   - 每个角度包含：建议标题、适合方向、核心问题、可搜集材料

4. **可继续追问的问题**
   - 给出 5-8 个后续问题
   - 问题要具体，不要泛泛而谈

## 选题偏好

优先选择能连接以下方向的新闻：

- 新闻传播
- 经济传播
- 公司报道
- 平台治理
- AI 产品与技术产业化
- 金融科技
- 资本市场和企业叙事
- 学术研究切口

## 信息质量要求

- 涉及最新新闻、市场、政策、公司动态时，必须使用最新可靠来源。
- 优先使用 Reuters、AP、Bloomberg、Financial Times、Wall Street Journal、公司公告、监管文件、官方数据等可信来源。
- 如果信息仍不确定，要明确说明。
- 不要把未经证实的市场传言写成事实。

## 语言风格

- 用中文。
- 尽量大白话，但不要幼稚化。
- 先讲清楚“这件事到底意味着什么”。
- 少用空泛判断，多讲因果。
- 不要像新闻摘要机器人一样堆条目。

## 避免

- 不要只罗列标题。
- 不要把所有新闻都说成“重大影响”。
- 不要给投资建议或承诺收益。
- 不要为了凑数量加入无关小新闻。
```

# 微信表情包处理 Skill

## 什么时候使用

当用户需要处理、切分、生成、压缩、检查微信表情包素材时，使用这个 Skill。

## 核心目标

把用户的表情包图片处理成更适合微信表情开放平台上传的素材，尤其注意尺寸、透明背景、文件大小、暗夜模式显示效果和裁剪完整性。

## 固定规格

### 单个表情

- 尺寸：240 × 240 px
- 背景：优先透明背景
- 文件大小：小于 500KB
- 格式：PNG 优先
- 要保留完整人物、文字、动作、装饰元素
- 不要裁掉边缘
- 不要把相邻表情裁进去

### Banner

- 尺寸：750 × 400 px
- 文件大小：尽量小于 80KB；如果用户另有要求，以用户要求为准
- 适合横向展示
- 字体清晰，不要太拥挤

### Icon

- 尺寸：50 × 50 px
- 文件大小：小于 100KB
- 通常只保留头部或最有识别度的主体
- 小尺寸下也要看得清

## 暗夜模式要求

- 角色和文字不能只有黑线透明底，否则暗夜模式可能看不清。
- 可添加白色贴纸边、浅灰底、白色填充或描边。
- 如果是异形抠图，边缘要干净，不能有明显毛边、透明残影或黑边。
- 文字不要过小，低饱和可以，但必须可读。

## 工作流程

1. 先确认用户上传的是：
   - 合集图
   - 单张表情
   - banner
   - icon
   - 还是整套素材
2. 如果是合集图：
   - 识别每个表情边界
   - 分别裁剪
   - 保留完整元素
   - 输出单图文件
3. 统一处理：
   - 调整到 240×240
   - 保持主体居中
   - 留合理边距
   - 添加透明背景或浅色底
   - 压缩文件大小
4. 检查：
   - 尺寸是否正确
   - 文件大小是否符合
   - 暗夜模式是否可见
   - 是否裁剪错误
   - 是否文字清晰
5. 输出：
   - 单个 PNG 文件
   - ZIP 压缩包
   - 简单说明文件

## 输出格式

优先输出可下载文件，并说明：

- 处理了多少张
- 每类素材尺寸
- 文件大小是否符合
- 是否保留透明背景
- 是否适配暗夜模式

## 避免

- 不要重新设计成完全不同角色，除非用户明确要求。
- 不要改变用户已经确认的风格。
- 不要把人物裁残。
- 不要只给说明不给下载文件。
- 不要忽视文件大小限制。
```

# 学术论文修改 Skill

## 什么时候使用

当用户需要修改论文、课程报告、文献综述、学术段落、参考文献、研究问题、标题、小标题时，使用这个 Skill。

## 核心目标

让文字更像真实学者写出来的文章：**学术但不僵硬，逻辑清楚，引用不乱配，少 AI 腔**。

## 固定风格要求

- 中文表达自然、准确、克制。
- 不要过度华丽。
- 不要空泛拔高。
- 保留用户原本的论点和写作意图。
- 如果用户说“微改”，只做最小必要修改。
- 如果用户说“不要大改”，不要重写结构。
- 优先改逻辑，再改语言。

## 引用与事实规则

- 不要编造参考文献。
- 不要把引用随便挪到不支持的句子后面。
- 每个实质性判断都要考虑是否需要出处。
- 如果不确定某个引用是否支持当前句子，要提醒“需要核实”。
- 不要为了通顺而改变事实范围。
- 不要把“可能”“一定程度上”改成过度确定的判断。

## 常见问题检查

修改时重点看：

1. 句子是否太长
2. 主语是否清楚
3. 概念是否跳跃
4. 前后逻辑是否接得上
5. 引用是否对应当前观点
6. 是否有空话套话
7. 是否像 AI 写的
8. 小标题是否准确概括内容
9. 结尾是否突然拔高
10. 是否有事实未经支持

## 输出模式

### 微改模式

当用户说“微改”“最小修改”“不要大改”时：

输出：
1. 修改后的版本
2. 简短说明改了哪里
3. 如有必要，提示引用风险

### 逻辑检查模式

当用户问“这段顺吗”“引用对吗”“有没有问题”时：

输出：
1. 主要问题
2. 建议修改方向
3. 可直接替换的句子
4. 需要核实的地方

### 标题/小标题模式

当用户要求标题建议时：

输出：
- 多个标题选项
- 每个标题的风格说明
- 哪个最稳妥
- 哪个更有文学性
- 哪个更适合新闻/学术

## 避免

- 不要把文章改成“AI 八股文”。
- 不要机械使用“首先、其次、最后”。
- 不要堆“多维度、赋能、机制、路径、格局”等空泛词。
- 不要为了显得高级而牺牲清楚。
- 不要擅自添加用户没有证据支撑的新观点。
```
